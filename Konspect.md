# Общий конспект
DevOps - главная роль - автоматизировать работу разработчиков и вывод продукта к пользователю быстро и без ошибок

CI/CD
CI (Непрерывная интеграция)- запускается после добавления разработчиком кода в репозиторий. Автоматически запускает проект, проводит тесты, ищет уязвимости. Если какие-то неполадки наблюдаются говорит об этом разработчику, если все ок - переходим к CD.
CD (Непрерывная доставка) - код автоматически отправляется в тестовую среду, а потом в копию прода. Чтобы протестировать в реальных условиях без серьезных угроз.

PIPELINE - инструкция как работает СI/CD. Четкий маршрут кода от компа до прода. Не просто набор команд, а визуальное отображение того на каком этапе код и как все поломалось + время выполнения


Iac (Инфраструктура как код)- составляем файл, в котором описываем окружение. Запускаем приложение (например TERRAFORM) и получаем окружение. Таким образом будем запускать его хоть 1000 раз и всегда один результат.

Config Management - запускаем приложение (например ANSIBLE) и описываем в нем в файле все что должно быть в окружении: все библиотеки и зависимости (nginx, curl, git и т.д). Запускаем и получаем этот конфиг на всех серверах запущенных Iac


КОНТЕЙНЕР - стандартизированный пакет с приложением и всем необходимым для его работы

МИКРОСЕРВИСЫ - разбиение большого приложения на маленькие независимые куски. Например каждый микросервис лежит в своем контейнере


ОРКЕСТРАЦИЯ - у нас есть множество контейнеров которыми надо управлять. Чтобы не делать этого вручную мы ставим оркестратор ( НАПРИМЕР KYBERNETES), прописываем в файлах настройки что мы хотим делать.

МОНИТОРИНГ - постоянное наблюдение за работой сервисов. Видим проблемы заранее

AGILE - Методология при которой задачи делятся на части

Scrum - Дополнение к Agile когда все задачи делятся на спринты(1-4 недели) и в конце каждого проводится анализ.

## Процессы в linux 

Процесс - исполняемая программа + контекст
У каждого процесса есть свой уникальный номер - PID.

TGID - общий номер для потоков внутри процесса

Допустим, есть процесс:

Process A (PID = 1000)

Он создаёт 3 потока:

Поток	 TID	TGID
main	 1000	1000
thread1	 1001	1000
thread2	 1002	1000

TID разные (каждый поток уникален)
TGID одинаковый (группа = один процесс)

Все процессы хранятся в hash таблице для быстрого поиска по PID

Самый главный процесс в Linux - systemd. 
Все процессы имеют своего родителя. Новые процессы появляются при помощи системного вызова fork - создает полную копию родителя. 
exec - пересоздает процесс с тем же самым pid

ps aux - снимок всех процессов в системе

kill pid - завершить процесс
kill -9 pid - принудительно завершить процесс

Когда процесс умирает пока родитель не вызовет wait() - он будет зомби-процессом(отображаться в таблице). Если процесс-родитель умрет раньше ребёнка то процесс-ребёнок(процесс-сирота) переходит к init и тот его хоронит.

Процессы в Linux изолированы друг от друга. Но есть способы межпроцессного взаимодействия.
1) Самый быстрый способ - разделяемая память (фрагмент памяти в которую имеют доступ несколько процессов)
Плюс - скорость
Минус - синхронизация
2) Pipes - труба для передачи данных. один процесс пишет во вход, другой читает на выходе
3) Сигналы - короткие сообщения. За передачу сигналов отвечает ядро
4) Сокеты - работают внутри одной машины. В отличии от пайпов процессы могут как писать так и читать.

## Метрики в Linux

Главная команда, которую тут надо знать - top, либо её продвинутая версия - htop
В выводе этих команд - таблица процессов в реальном времени и набор важных параметров, которые СРАЗУ дают понимание, что происходит в системе:
load average - средняя нагрузка на систему за последние 1, 5, 15 минут.
us - время CPU на пользовательские процессы
sy - время CPU на системные вызовы
wa - I/O Wait, то есть CPU ждёт ответа от диска
sі - системные прерывания
id - простой CPU

## Работа с диском

Комбинация из работы с командами df и du

df -h - вывести все разделы диска и их размеры

du -sh * | sort -hr - вывести все папки текущей директории + отсортировать их по размеру

Но иногда df показывает что место на диске есть, а система говорит что нет. Это значит что закончились INODE(записи на диске), а бывает такое когда лежит огромное количество очень мелких файлов.
На сервере любой диск/флешку при подключении надо примонтировать командой mount
lsblk - посмотреть подключенные диски в /dev
sudo mount /dev/sbp /data - примонтировали устройство sbp к нашей файловой системе в папку /data

1.Обновление системы
sudo apt update && sudo apt upgrade -y
Что делает:
sudo → запускает с правами администратора
apt update → обновляет список пакетов (НЕ устанавливает, только проверяет что нового есть)
&& → означает “выполни вторую команду только если первая успешна”
apt upgrade → устанавливает обновления
-y → автоматически отвечает “yes” на вопросы

Итог: система обновлена и безопасна

2.Создание пользователя
sudo adduser serveradmin
Что делает:
создаёт нового пользователя serveradmin

создаёт группу с таким же именем
спрашивает пароль

Итог: отдельный пользователь вместо root

3.Добавление в sudo-группу
sudo usermod -aG sudo serveradmin
Разбор:
usermod → изменить пользователя

serveradmin → кого добавляем

Итог:пользователь получает право использовать sudo

Что делает:

показывает, в каких группах состоит пользователь

важно: проверка что sudo реально выдан

su - serveradmin
Что делает:
su → switch user (смена пользователя)

Итог: ты становишься serveradmin

Что происходит:
whoami → показывает текущего пользователя

### Написать свой systemd- юнит

1.Создаём папку

Под пользователем serveradmin:

mkdir -p ~/myserver

cd ~/myserver

2.Создаём простой сервер
nano server.py

Содержимое:

from http.server import HTTPServer, SimpleHTTPRequestHandler

server = HTTPServer(("0.0.0.0", 8000), SimpleHTTPRequestHandler)

print("Server started on port 8000")

server.serve_forever()

Проверка:

python3 server.py

В браузере:

http://192.168.100.129:8000

Должна открыться страница.

Остановить:

Ctrl+C
3.Создаём systemd unit
sudo nano /etc/systemd/system/myserver.service

Содержимое:

[Unit]
Description=My Python HTTP Server
After=network.target

[Service]
Type=simple
User=serveradmin
WorkingDirectory=/home/serveradmin/myserver

ExecStart=/usr/bin/python3 /home/serveradmin/myserver/server.py

Restart=always
RestartSec=3

MemoryMax=100M

[Install]
WantedBy=multi-user.target
Разбор
After=network.target

Сначала сеть → потом сервис.

User=serveradmin

Запускать НЕ от root.

WorkingDirectory

Текущая папка процесса.

ExecStart

Что запускать:

/usr/bin/python3 server.py
Restart=always

Если процесс завершится:

systemd запустит его снова
RestartSec=3

Подождать 3 секунды перед рестартом.

MemoryMax=100M

Ограничение памяти через cgroups.

Больше 100 МБ процесс использовать не сможет.

4.Перечитать конфигурацию

После создания юнита:

sudo systemctl daemon-reload

Что происходит?

systemd перечитывает:

/etc/systemd/system
5.Запуск сервиса
sudo systemctl start myserver

Проверка:

systemctl status myserver

Должно быть:

Active: active (running)
6.Автозапуск
sudo systemctl enable myserver

Теперь после перезагрузки сервис поднимется сам.

Проверка:

systemctl is-enabled myserver

Ответ:

enabled
7.Проверка через браузер

Открой:

http://192.168.100.129:8000

Если страница открылась — сервис работает.

8.Проверка рестарта после kill -9

Найти PID:

systemctl status myserver

или

ps aux | grep server.py

Например:

PID 4321

Убиваем:

sudo kill -9 4321

Через пару секунд:

systemctl status myserver

Увидишь новый PID.

Это означает:

systemd заметил смерть процесса
↓
запустил новый

9.Проверка лимита памяти

Посмотреть настройки:

systemctl show myserver | grep Memory

Будет что-то вроде:

MemoryMax=104857600

100 МБ.

10.Посмотреть логи

Очень важная команда:

journalctl -u myserver -f

Показывает лог сервиса в реальном времени.

Если сервер упал:

systemd restart service

это будет видно здесь.
